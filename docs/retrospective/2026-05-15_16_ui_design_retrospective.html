5. 근본 원인 분석

spacing 12 commits 실패의 7 가지 근본 원인

1

시각 검증 사이클 부재

코드 변경 → push → 사용자 확인 → 화남 → 다시. 한 사이클 15~30분. 12 commits = 12 사이클.
만약 AI 가 30초 안에 시각 검증했다면 1-2 commits 안에 정답.

2

그림 의미 매번 다른 해석

사용자 화살표 "외곽 = 사이 동일" 을 6 번 다르게 해석. 4차에서 정반대 해석.
디자인 원칙 변환 없이 직관 추측만 반복.

3

사용자 시각 임계값 무시

4px 변화 (cdea382) 는 인식 불가능. 일반 사용자 시각 임계값은 8~16px 단위.
변화 폭 가이드라인 부재.

4

Visual hierarchy 무시

spacing 자체만 보고 배경색/border/shadow 같은 visual boundary 요소는 검토 안 함.
10차 commit 에서야 카드 bg/border 검토 — 너무 늦음.

5

서브에이전트 위임 늦음

6차 (380b5c4) 에서야 ui-designer 위임. 그것도 spacing 만 손대고 visual boundary 미검토.

6

worktree deps 부재

git worktree + monorepo + 외부 .env + dist 결합 → 셋업 비용 큼. 1~2 시간 낭비.

7

CI 결과 확인 안 함

세션 시작 ~5 commits 까지 CI failure 인지 모르고 진행. 사용자가 직접 GitHub 화면 보낸 후 발견.

6. 개선 방안

단기 (다음 세션 즉시)

A. 시각 검증 자동화 셋업 필수

목표: 코드 변경 → 30초 안에 AI 가 직접 시각 확인.

# scripts/visual-check.sh pnpm --filter @storywork/web dev --port 3000 & sleep 10 chromium --headless --window-size=1280,800 \ --screenshot=./tmp/pose-panel.png \ http://localhost:3000/editor

또는 Playwright snapshot 테스트로 모든 panel 자동 비교.

B. 디자인 토큰 단일 source

/* apps/web/app/editor/editor.css */ :root { /* spacing — 외곽 = 그리드 gap 항상 동일 */ --editor-panel-pad: 16px; --editor-panel-gap: var(--editor-panel-pad); /* 카드 시각 분리 (panel boundary 명확화) */ --editor-card-bg: var(--color-surface); /* white */ --editor-card-border: 2px solid var(--color-border-strong); --editor-card-shadow: 0 1px 2px rgba(0,0,0,0.04); }

모든 panel 에서 이 토큰만 사용. 변경 시 한 곳만 손댐.

C. ESLint 룰: panel 1:1 비율 강제

panel grid 의 p- 와 gap- 동일 값 강제. 위반 시 lint fail → 회귀 자동 차단.

중기 (1~2 주)

D. 의도 → 명세 변환 워크플로우

[사용자] "다닥다닥" [AI 명세표] 현재: padding 16px, gap 16px, card bg 같은 색 진단: visual boundary 부족 제안: padding 16 유지, card bg white + border-2 추가 [사용자 동의/거부] [AI 구현]

E. 시각 변화 인식 임계값 가이드

  • padding / gap: 8px 이상 변화 (4px 변화는 인식 불가능)
  • font size: 2px 이상
  • color: hue 차이 명확 (#fff vs #f5f5f5 는 거의 동일)
  • border: 1px → 2px 차이 명확
  • shadow: 0 → shadow-sm 차이 명확

장기 (1~2 개월)

F. Storybook + Chromatic visual regression

모든 panel/card 컴포넌트의 visual snapshot 자동 비교. 변경 시 diff 자동 표시.

G. 디자인 시스템 문서 자동 생성

DESIGN-nike.md 의 implementation_contract 가 코드 invariant test 로 자동 검증. 토큰 정의 → docs 자동 빌드.

7. 바이브 코딩 협업 방법론 제안

핵심 원칙: "사용자는 의도를 말한다, AI 는 명세화 + 구현 + 검증한다."
사용자가 수치를 직접 줘야 한다면 그것은 바이브 코딩이 아니다.

8 가지 원칙

1

의도 → 명세 변환 의무

사용자 직관 표현 → AI 가 즉시 명세표 만들어 동의 받은 후 구현. 추측 금지.

2

시각 변화는 항상 직접 검증

UI 작업은 AI 가 screenshot 으로 직접 확인 후 push. 메인 디렉토리 dev + headless screenshot 필수.

3

변화 폭은 인식 임계값 이상

4px 변화는 의미 없음. 항상 8px+, 또는 색/border/shadow 명확 변화.

4

2 번째 실패 즉시 서브에이전트

같은 시도 2 번째 실패 시 무조건 ui-designer 등 전문 위임. 추측 반복 금지.

5

Visual hierarchy + spacing 함께

spacing 만 손대지 말 것. background / border / shadow / typography 함께 검토.

6

CI 결과 매 push 즉시

push 후 GitHub Actions polling 자동 시작. 실패 시 다음 작업 전 fix.

7

CSS chunk hash 모니터

UI 변경 push 후 prod CSS hash 변하지 않으면 사용자 캐시 hit 가능. arbitrary value 또는 incognito 안내.

8

3 번째 실패 = 즉시 멈춤

같은 commit 패턴 3 번 실패 시 무조건 멈추고 회고. 추측 push 반복 금지. 사용자 시간/토큰 보호.
